# BQA-OS 2-Week QA Memory Pilot

## Who It Is For

QA Leads and QA Automation Leads in teams with many recurring QA decisions,
flaky handoffs, or hard-to-reuse test knowledge.

Best fit:
- B2B SaaS, fintech, and data-heavy platform teams.
- Teams with existing test runs, bug reports, release notes, or QA checklists.
- Teams that want a local-first assistant workflow before adopting a wider
  agent platform.

## Problem

QA knowledge often lives in scattered tickets, chat threads, ad hoc scripts, and
one person's memory. Each new release repeats the same context gathering:
what changed, what needs regression, what data is safe, what failed before, and
which checks prove readiness.

## Pilot Offer

In two weeks, BQA-OS turns a focused set of QA evidence into a reusable QA
memory pack:
- role prompts for QA, architecture, safety, and implementation support;
- repeatable test workflows;
- test spec templates and data-quality checklists;
- validation reports and drift guardrails;
- a concise next-step report for the team.

## Inputs Needed

Use public-safe or sanitized artifacts:
- test plans or checklists;
- issue templates and acceptance criteria;
- synthetic session examples;
- sanitized validation reports;
- current repo commands for tests and lint.

Do not provide:
- credentials;
- raw customer data;
- production dumps;
- private logs that have not been sanitized.

## Outputs

The pilot produces:
- local QA memory pack;
- recommended test and validation workflows;
- onboarding notes for QA engineers;
- risks and missing evidence report;
- demo-ready synthetic archive and browser view when relevant.

## Success Criteria

The pilot is useful if the QA team can:
- run or adapt at least one generated workflow;
- reuse at least one generated test spec;
- identify missing validation evidence faster than before;
- explain the safety boundaries to engineering and security stakeholders.

## Call To Action

Pick one release, one module, or one ETL pipeline. Run BQA-OS on sanitized
evidence, review the generated QA memory pack, and decide whether to expand.
